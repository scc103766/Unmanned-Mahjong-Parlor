## 2. 产品目标

### 2.1 用户价值
- 顾客：24小时自助预约房间、支付、开门，无需人工前台
- 商家：降低人力成本，远程管理门店、房间、价格、订单、设备、保洁、财务
- 保洁员：通过任务流清晰接收清洁任务、开门权限、上传凭证、获得结算
- 平台运营方：多租户管理、设备供应商接入、异常监控与审计

### 2.2 产品边界（不可逾越）
- ❌ 前端不作为权限/库存/支付/设备控制的最终可信来源
- ❌ 不可跨租户访问数据
- ❌ 不可重复售卖同一房间同一时段
- ✅ 允许商家自主配置门店、房间、价格、优惠、设备

## 3. 技术架构

### 3.1 总体管线
```
顾客H5/小程序/App → FastAPI 后端 → PostgreSQL + Redis
                          ├── 微信支付
                          ├── 硬件门锁/门禁/电源适配器
                          └── 对象存储 (MinIO/S3)
商家Web后台 → 同一后端 API
平台Web后台 → 同一后端 API
```

### 3.2 技术栈
| 层次 | 方案 |
|------|------|
| 后端 | FastAPI + Pydantic v2 + SQLAlchemy 2.x + Alembic |
| 数据库 | PostgreSQL (生产) / SQLite+aiosqlite (开发) |
| 缓存/幂等 | Redis |
| Web前端 | Vue 3 + Vite + TypeScript + Pinia + Vue Router |
| 小程序 | 微信原生小程序 (参考项目 24h_qipaishi) |
| App | Flutter (骨架已搭建，后续移植) |
| 测试 | pytest + httpx + ruff + mypy |
| 部署 | Docker Compose |

## 6. 角色体系

### 6.0 用户（你）— 最终决策者 & 学习者
- 角色：项目所有者、最终把关人、学习者
- 权限：**所有决策必须经你批准才能推进**
- 权利：对任何方法/工具有完整知情权、否决权、审批权

### 6.1 PM Agent (DeepSeek V4 / pi) — 翻译官 & 把关前置
- 三重角色：产品经理 + 技术教师 + 把关前置
- 核心约束：**未经你明确批准，不得向Engineer下发任何任务**
- 标注规则：推理/设计/教学/审核时标注 🎯 DeepSeek V4 PM

### 6.2 Engineer Agent (Codex / GPT-5.5) — 执行者
- 职责：代码编写、文件操作、shell执行
- 核心约束：只执行已批准的Task，不得自行变更需求
- 工作模式：**非交互式**，接收任务→执行→产出代码+开发日志
- 标注规则：代码执行时标注 🔧 Codex Engineer
- 通信方式：通过 `tasks/` 文件队列

### 6.3 审批与验收
- 🎯 PM 自审后 → 提交你审批 → 你明确"批准"后 → 下发 Codex
- 🔧 Codex 完成后 → PM 解读 → 🎯 PM + 你共同验收
- 验收通过 → 更新 PROJECT_CONTEXT.md → 进入下一任务
