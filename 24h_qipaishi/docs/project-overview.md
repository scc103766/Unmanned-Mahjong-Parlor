# 现有参考项目概览

## 项目定位

`24h_qipaishi` 是一个“麻老板无人自助系统”微信小程序端参考实现，服务于 24H 自助棋牌室、共享茶室、台球室、私人影院等无人值守空间。它的核心用户路径是：顾客选择门店和房间，按时间段预约，下单支付，到店后通过小程序开门/开电，消费后触发保洁任务，商家在小程序内管理门店、房间、订单、优惠和经营数据。

## 仓库类型

- 仓库形态：单体小程序前端项目。
- 项目类型：微信原生小程序。
- 后端依赖：远程 REST API，当前 baseUrl 为 `https://wq.scyanzu.com/app-api`。
- 硬件依赖：门锁/门禁/电源/云喇叭等由后端 API 或微信插件间接控制。
- 不包含内容：服务端代码、数据库 schema、硬件接入服务、Web 管理后台。

## 技术栈

| 类别 | 技术 | 当前位置 | 说明 |
| --- | --- | --- | --- |
| 小程序框架 | 微信原生小程序 | `app.js`, `app.json`, `pages/**` | JS/WXML/WXSS 页面式组织 |
| UI 组件 | Vant Weapp `^1.10.22` | `package.json`, `miniprogram_npm/@vant/weapp` | 表单、弹窗、日历、上传、下拉、标签等 |
| 图表 | ECharts + ec-canvas | `ec-canvas/**`, `pages/statics` | 经营统计、订单统计、房间使用率等 |

| 网络层 | 自定义封装 | `utils/http.js` | 统一 baseUrl、tenant-id、Bearer token、401 跳转 |
| 硬件插件 | 微信插件 `myPlugin` | `app.json`, `utils/lock.js` | 蓝牙门锁本地开锁 |

## 页面规模

当前 `pages` 下共有 47 个页面 JS 文件。业务页面大致可分为：

- 顾客端：首页、门店列表、房间列表、预约、订单提交、订单详情、我的订单、充值、优惠券、余额、个人资料、帮助。
- 门店/房间管理：门店列表、门店设置、房间列表、房间设置、房间开关门、二维码、云喇叭测试。
- 订单管理：订单列表、订单改期/续费/取消/退款、换房、管理员订单处理。
- 保洁管理：保洁账号、任务列表、任务详情、任务接单/开始/完成、结算、保洁统计。
- 营销财务：优惠券、会员、充值赠送规则、团购码核销、提现、经营统计。

## 关键业务能力

1. 顾客预约：按门店、房间、日期、时长、通宵场、优惠券、团购码生成订单。
2. 支付能力：支持微信支付、余额支付、团购码/优惠券抵扣。
3. 订单履约：订单详情页支持确认到店、续费、取消、换房、分享订单、一键开门。
4. 硬件控制：通过订单或管理接口控制大门、房间门、远程锁；蓝牙锁作为部分场景兜底。
5. 保洁闭环：订单后保洁任务可接单、开始、开门、完成、投诉与结算。
6. 商家运营：门店/房间配置、价格规则、优惠券、会员、提现和经营图表。
7. 多租户雏形：请求头包含 `tenant-id`，但前端仍有硬编码租户和接口地址。

## 主要风险与限制

- 前端保存 baseUrl、tenantId 等环境信息，缺少环境隔离和配置治理。
- 订单、支付、开门等关键链路强依赖后端幂等与权限校验，当前小程序端无法证明后端实现。
- 管理端能力混在小程序内，顾客端、保洁端、商家端边界不够清晰。
- 硬件失败、网络失败、支付回调延迟、蓝牙开锁失败等异常路径需要系统化设计。
- 当前仓库缺少自动化测试、类型约束和 API 契约文档。
