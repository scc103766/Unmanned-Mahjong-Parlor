# PRD: 24H 无人棋牌室管理系统

## 1. 产品目标

建设一套面向 24 小时无人棋牌室/共享茶室的管理系统，覆盖顾客预约消费、商家门店管理、设备开门开电、保洁履约、支付结算、营销和运营统计。产品要从参考小程序的功能经验出发，形成可扩展、可审计、可多租户部署的完整业务系统。

### 业务目标

- 支持门店无人或少人值守营业。
- 提升顾客自助预约、支付和开门体验。
- 降低商家人工开门、排班、清洁协调和对账成本。
- 支撑多门店、多租户、连锁化运营。
- 让硬件控制、支付、订单和保洁形成闭环，降低运营风险。

### 用户目标

- 顾客能快速找到合适房间、明确价格、完成支付并可靠开门。
- 商家能自主配置门店、房间、价格、优惠、设备和运营数据。
- 保洁员能清楚接收任务、开门、完成清洁并获得结算。
- 平台运营能管理租户、权限、设备接入、异常和审计。

## 2. 角色与权限

| 角色 | 核心诉求 | 关键权限 |
| --- | --- | --- |
| 顾客 | 预约房间、支付、开门、续费、取消、查看订单 | 仅操作自己的订单和授权订单链接 |
| 商家/店长 | 管理门店、房间、价格、订单、优惠、统计和提现 | 管理所属租户/门店 |
| 店员/管理员 | 处理订单、退款、换房、异常开门 | 按门店授权 |
| 保洁员 | 接任务、开门清洁、上传完成凭证 | 仅操作分配/可接任务 |
| 平台运营 | 租户管理、系统配置、设备供应商、审计 | 跨租户平台级权限 |
| 客服 | 协助顾客处理开门失败、支付异常、退款争议 | 受控查看订单和执行补救动作 |

## 3. 业务流程

### 3.1 顾客预约消费流程

1. 顾客进入小程序。
2. 系统定位或让顾客选择城市/门店。
3. 顾客查看门店公告、地址、Wi-Fi、房间图片和可预约时间轴。
4. 顾客选择房间、开始时间、消费时长或通宵场。
5. 系统根据房间价格、工作日价、通宵价、最小时长、优惠券、团购码计算金额。
6. 顾客提交预订单。
7. 顾客选择微信支付、余额支付、优惠券抵扣或团购码。
8. 支付成功后生成正式订单并锁定房间时间段。
9. 到店后顾客在订单详情页确认到店并开大门/房间门/智能锁/电源。
10. 消费中可续费或联系商家。
11. 订单结束后系统触发清洁任务，房间进入待清洁或可配置状态。

### 3.2 商家管理流程

1. 商家创建或选择门店。
2. 配置门店名称、地址、坐标、客服、公告、Wi-Fi、图片、清洁时间、通宵参数。
3. 配置房间：房间类型、标签、图片、价格、工作日价、通宵价、最小时长、提前预约、不可用时段、设备配置、云喇叭文案。
4. 配置优惠券、充值赠送规则、会员发券、团购码核销。
5. 查看订单，处理取消、退款、换房、改时、续费。
6. 查看经营统计，申请提现。

### 3.3 保洁流程

1. 订单结束、提前结束或人工触发后生成保洁任务。
2. 保洁员按门店和状态筛选任务。
3. 保洁员接单并到店。
4. 系统按任务授权保洁员开大门和房间门。
5. 保洁员开始任务、上传现场/完成图片。
6. 商家或系统确认完成，任务进入待结算/已结算。
7. 异常任务可投诉、驳回或人工处理。

### 3.4 设备控制流程

1. 前端发起开门/开锁/开电请求。
2. 后端校验操作者、角色、订单或任务、时间窗口、门店和房间。
3. 后端生成设备命令并调用设备供应商。
4. 系统记录命令、响应、失败原因和重试。
5. 前端展示操作结果。
6. 对支持蓝牙锁的房间，可在远程失败时提示本地蓝牙开锁兜底。

## 4. 功能需求

### 4.1 顾客端

- FR-C-001：用户可通过微信手机号授权登录。
- FR-C-002：用户可按城市/定位选择门店。
- FR-C-003：用户可查看门店详情、公告、客服、Wi-Fi、地图和环境图片。
- FR-C-004：用户可查看房间列表、图片、标签、价格、可预约时间轴。
- FR-C-005：用户可选择开始时间、时长、通宵场和其他时长。
- FR-C-006：系统应展示订单原价、优惠后金额、可用优惠券数量、余额。
- FR-C-007：用户可使用微信支付、余额、优惠券、团购码完成下单。
- FR-C-008：用户可查看订单列表和订单详情。
- FR-C-009：用户可在授权时间内确认到店、开大门、开房间门、开智能锁或开电。
- FR-C-010：用户可续费、取消订单、换房和分享订单。
- FR-C-011：用户可充值余额并查看余额/赠送余额流水。
- FR-C-012：用户可查看使用帮助和联系客服。

### 4.2 商家管理端

- FR-M-001：商家可管理门店基础资料、图片、公告、客服、Wi-Fi、地图坐标。
- FR-M-002：商家可配置门店通宵开始时间、通宵时长、清洁时间、是否允许未清洁预约。
- FR-M-003：商家可管理房间信息、分类、规格、标签、图片、排序。
- FR-M-004：商家可配置房间普通价、工作日价、通宵价、最小时长、提前下单天数/小时。
- FR-M-005：商家可配置房间不可预约时段。
- FR-M-006：商家可维护设备绑定信息，包括门禁、房间门、智能锁、电源、云喇叭。
- FR-M-007：商家可查看订单，执行取消、退款、改时、续费、换房。
- FR-M-008：商家可直接开大门、房间门、智能锁、关门或禁用房间。
- FR-M-009：商家可管理优惠券、发券、充值赠送规则、会员。
- FR-M-010：商家可核销团购券码。
- FR-M-011：商家可查看收益、订单、会员、充值、房间使用率和房间使用时长统计。
- FR-M-012：商家可申请提现并查看提现记录。

### 4.3 保洁端

- FR-K-001：保洁员可查看可接、已接、已开始、已完成、已取消、被驳回、已结算任务。
- FR-K-002：保洁员可接单、取消接单、开始任务、完成任务。
- FR-K-003：保洁员可在任务授权下开大门和房间门。
- FR-K-004：保洁员完成任务时应上传清洁凭证。
- FR-K-005：商家可投诉、审核、结算保洁任务。
- FR-K-006：系统应提供保洁数据统计和账单。

### 4.4 平台与租户

- FR-P-001：系统支持多租户，每个租户可拥有多个门店。
- FR-P-002：租户数据必须隔离，前端传入的 `tenant-id` 不能作为唯一可信来源。
- FR-P-003：平台可配置设备供应商、微信小程序、支付商户号、对象存储。
- FR-P-004：平台可查看跨租户系统异常、设备异常、支付异常和审计日志。

## 5. 非功能需求

| 类别 | 要求 |
| --- | --- |
| 可用性 | 顾客下单、支付、开门链路需 24 小时可用；核心接口需监控和告警 |
| 响应时间 | 普通页面接口 P95 小于 1 秒；开门命令 P95 小于 3 秒返回可解释状态 |
| 安全 | 设备控制、退款、提现、结算必须强鉴权和审计；订单分享 key 必须有权限边界 |
| 幂等 | 支付回调、订单保存、续费、退款、设备命令必须幂等 |
| 数据一致性 | 房间时间段不可超卖；支付状态与订单状态最终一致 |
| 可观测性 | 记录订单、支付、设备命令、保洁任务、异常和人工操作日志 |
| 隐私 | 手机号、定位、订单、支付信息按最小必要原则展示和存储 |
| 扩展性 | 设备供应商、支付渠道、营销玩法、门店业态可扩展 |
| 兼容性 | 支持主流微信版本；弱网下给出清晰失败提示和重试路径 |

## 6. 关键数据实体

- Tenant：租户、套餐、配置。
- Store：门店、地址、坐标、公告、客服、Wi-Fi、图片、营业参数。
- Room：房间、分类、规格、价格、工作日价、通宵价、标签、图片、状态。
- RoomAvailability：可预约时间、禁用时段、锁定时段。
- Order：订单、时间、金额、状态、orderKey、支付方式、房间、用户。
- Payment：支付单、微信支付参数、回调状态、退款状态。
- BalanceAccount：余额、赠送余额、流水。
- Coupon：优惠券定义、领取记录、使用记录。
- GroupBuyCode：团购码、核销记录。
- Device：设备、供应商、门店/房间绑定、在线状态。
- DeviceCommand：命令、操作者、订单/任务、响应、失败原因。
- CleaningTask：保洁任务、状态、保洁员、凭证、结算。
- Withdrawal：提现申请、审核、到账状态。
- AuditLog：关键操作审计。

## 7. 技术边界

### 小程序端

- 负责展示、交互、微信授权、微信支付发起、定位、扫码、图片上传、蓝牙锁本地控制。
- 不负责最终权限判断、库存锁定、订单状态裁决和设备安全校验。

### 服务端

- 负责租户、用户、权限、门店、房间、订单、支付、优惠、余额、保洁、统计。
- 负责设备命令编排、幂等、重试、审计和状态回写。
- 负责与微信支付、硬件供应商、对象存储等第三方集成。

### 硬件层

- 设备供应商差异通过适配器隐藏。
- 前端只感知“可开门/不可开门/操作中/成功/失败原因”。
- 对断网、离线、超时、重复命令、设备状态不同步要有统一策略。

### 管理后台

- MVP 可复用小程序管理页，但中长期建议提供 Web 管理端。
- 平台级租户管理、设备供应商管理、异常处理更适合 Web 后台。

## 8. MVP 范围

### 必须包含

- 顾客小程序完整预约支付开门流程。
- 门店/房间配置、价格规则、设备绑定。
- 订单管理、取消、续费、换房、退款入口。
- 微信支付、余额支付、优惠券、团购码。
- 保洁任务闭环。
- 经营统计基础版。
- 设备控制服务与至少一种门锁/门禁/电源适配。
- 审计日志和异常记录。

### 暂不包含

- 原生 App。
- 复杂营销自动化。
- 完整财务 ERP。
- 多硬件供应商全量适配。
- 大规模 IoT 实时监控大屏。

## 10. 验收标准

- 顾客可在测试门店完成从选房到支付再到开门的完整闭环。
- 同一房间同一时间段不能被重复预约。
- 支付成功、支付失败、支付回调延迟均能得到正确订单状态。
- 非订单用户、过期订单、越权管理员、非任务保洁员不能开门。
- 开门命令有完整日志：操作者、角色、设备、订单/任务、时间、结果。
- 订单结束后能自动产生保洁任务，保洁完成后房间可重新预约。
- 商家可配置门店、房间、价格、优惠和查看基础统计。
- 多租户数据隔离通过测试。

## 11. 风险与开放问题

- 需要确认目标硬件供应商、协议、回调机制和离线策略。
- 需要确认是否继续使用参考服务端，还是重建后端。
- 需要明确管理端是否继续放在小程序内，还是新增 Web 管理后台。
- 需要明确团购平台、支付商户号、提现方式和财务对账规则。
- 需要补充详细退款规则、异常开门补偿规则、保洁结算规则。
- 需要确认真实门店网络条件、设备在线率和应急人工流程。
