## 参考项目能力

参考项目是微信原生小程序前端，覆盖 24H 无人棋牌室/共享茶室的大部分业务场景：

- 顾客端：登录、定位选店、房间浏览、预约、支付、订单详情、开门、开电、续费、取消、换房、优惠券、余额、团购码。
- 商家端：门店资料、房间配置、价格、不可预约时段、设备绑定、订单处理、会员、优惠券、充值规则、提现和统计。
- 保洁端：任务列表、接单、开始、开门、上传凭证、完成、投诉、结算和统计。
- 设备端：店门、房间门、电源、远程门锁、蓝牙锁、云喇叭。
- 第三方：微信登录、微信支付、地图、图片上传、门锁网关。

## 参考项目主要问题

- 业务规则大量散落在小程序页面，价格、库存、订单状态和设备权限难以测试。
- `tenant-id` 从前端传入，不能作为多租户隔离的唯一可信依据。
- API 路径散落在页面文件中，缺少明确契约和版本策略。
- 开门、开电、退款、提现等高风险操作需要更完整的服务端审计。
- 支付回调、设备命令、订单续费和退款需要明确的幂等设计。
- 房间时段锁定、清洁时间、通宵价和不可用时段应进入服务端领域模型。

## Python 版建设目标

- 功能对齐参考项目，覆盖顾客、商家、保洁、平台运营四类角色。
- 服务端承接核心业务裁决，前端仅做展示和交互。
- 以 Python API 服务为中心，优先支持顾客 H5、保洁移动网页、商家 Web 后台和平台 Web 后台；微信小程序作为参考项目和可选兼容来源，App 作为后续移植方向。
- 从第一版开始提供多租户、RBAC、审计、幂等、可观测性和自动化测试。
- 在设备控制上使用命令表和供应商适配器，让硬件失败可追踪、可补偿。

## 需要优先保护的业务链路

1. 预约下单：房间时段不可超卖，价格计算可解释。
2. 支付回调：微信支付回调可重复处理，订单状态最终一致。
3. 到店开门：操作者、订单、任务、门店、房间、时间窗口全部校验。
4. 续费换房：不能破坏已锁定时段，金额和优惠规则清晰。
5. 订单结束：自动触发清洁任务，房间状态可配置为待清洁或可继续预约。
6. 退款提现：所有资金操作有权限、状态机、审计和对账依据。
