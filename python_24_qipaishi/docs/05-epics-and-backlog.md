# Epics 与 Backlog

## Epic 1：项目基础设施

目标：建立可持续开发的 Python 服务骨架。

Stories：

- 初始化 FastAPI 应用、配置管理和健康检查。
- 接入 PostgreSQL、Redis、SQLAlchemy、Alembic。
- 建立统一错误码、响应格式和 request_id。
- 建立 pytest、ruff、mypy 和基础 CI 命令。
- 生成 OpenAPI 并按业务域分组。

验收：

- 本地一条命令可启动 API。
- 测试和 lint 可运行。
- 数据库迁移可创建基础结构。

## Epic 2：租户、用户和权限

目标：所有业务能力在可靠的数据作用域内运行。

Stories：

- 建立 Tenant、TenantApp、User、Role、UserStoreScope。
- 实现微信登录 mock 和真实 provider 接口。
- 实现 JWT 登录态和权限依赖。
- 实现平台、商家、店员、保洁、顾客角色。
- 所有业务查询默认过滤 `tenant_id`。

验收：

- 不同租户数据不能互相访问。
- 无权限用户不能操作门店、订单、设备和资金接口。

## Epic 3：门店、房间和价格配置

目标：支撑商家配置完整经营资料。

Stories：

- 门店基础资料、图片、公告、客服、Wi-Fi、地图坐标。
- 门店通宵参数、清洁时间、是否允许未清洁预约。
- 房间资料、分类、标签、排序、图片。
- 房间普通价、工作日价、通宵价、最小时长。
- 房间不可预约时段和设备绑定。

验收：

- 可以完整创建一个门店和多个房间。
- 顾客端可以读取房间列表和价格摘要。

## Epic 4：可用性、预约和订单

目标：完成预约下单的核心交易链路。

Stories：

- 房间时间轴查询。
- 价格试算和明细返回。
- 预订单创建和待支付锁定。
- 正式订单创建和状态机。
- 订单列表、详情、取消、续费、换房。
- 待支付超时释放库存。

验收：

- 并发下单不能造成同房间同时间段重复占用。
- 续费和换房会重新校验库存。
- 订单状态流转符合状态机。

## Epic 5：支付、余额和营销

目标：完成资金和优惠闭环。

Stories：

- 微信支付预下单、回调、查单、关闭订单。
- 退款申请、退款回调、退款流水。
- 余额账户、充值、赠送余额、流水。
- 优惠券模板、领取、锁定、使用、退回。
- 团购码验证和核销。

验收：

- 支付回调可重复处理且只确认一次订单。
- 余额和优惠券发生变化时有完整流水。
- 退款金额不会超过可退金额。

## Epic 6：设备控制

目标：让开门、开锁、开电可审计、可追踪、可重试。

Stories：

- 建立 Device 和 DeviceCommand。
- 实现设备供应商接口和 mock 适配器。
- 实现订单授权开店门、开房门、开锁、开电。
- 实现商家和保洁授权开门。
- 实现命令状态查询、失败原因和重试。

验收：

- 每次设备操作都有命令记录和审计。
- 越权、超时段、订单取消后不能开门。
- mock 适配器可支持自动化测试。

## Epic 7：保洁任务

目标：订单结束后自动进入清洁履约闭环。

Stories：

- 订单结束生成清洁任务。
- 保洁员查看、接单、开始、开门、完成。
- 上传清洁凭证。
- 商家审核、驳回、投诉、结算。
- 清洁状态影响房间可预约状态。

验收：

- 一个订单不会重复生成多个有效保洁任务。
- 保洁员只能操作自己授权的任务。
- 任务完成后房间状态正确更新。

## Epic 8：商家运营和统计

目标：提供商家日常经营所需后台能力。

Stories：

- 订单后台处理：取消、退款、改时、续费、换房。
- 会员列表、消费记录、优惠券发放。
- 充值规则和提现记录。
- 收入、订单、退款、房间使用率、会员、保洁统计。
- 平台异常、审计和人工补偿入口。

验收：

- 商家可完成日常经营操作。
- 统计口径可追溯到订单、支付、退款和流水事实表。

## Epic 9：网页端

目标：交付顾客 H5、保洁移动网页、商家 Web 后台和平台 Web 后台。

Stories：

- 创建 Vue/Vite/TypeScript Web 工程、路由、主题、环境配置和 API client。
- 实现顾客 H5 预约、支付、订单、开门和续费取消流程。
- 实现保洁移动网页任务、接单、开门、凭证和结算记录。
- 实现商家 Web 后台工作台、门店房间、价格、设备、订单、保洁、财务和统计。
- 实现平台 Web 后台租户、权限、设备供应商、异常和审计。
- 建立 Web 构建、预览和 Playwright smoke 流程。

验收：

- 顾客 H5 可以完成预约支付开门闭环。
- 保洁移动网页可以完成现场履约和异常处理。
- 商家 Web 后台可以完成日常经营配置、订单处理和统计查看。
- 平台运营可以通过 Web 后台处理租户、异常和审计。
- Web 构建产物可部署。

## 首轮 Sprint 建议

周期：5 个工作日。

目标：完成 M0 项目基础，并为 M1 打开路。

任务：

1. 初始化项目结构和依赖管理。
2. 创建 FastAPI app、配置、健康检查。
3. 接入 SQLAlchemy、Alembic、PostgreSQL。
4. 接入 Redis 客户端和 request_id 中间件。
5. 建立统一错误和响应格式。
6. 创建 Tenant、User、Role 的首批迁移草案。
7. 写健康检查和基础数据库测试。
8. 创建 Web 工程骨架和 API client 草案。

完成标准：

- `GET /health` 可返回 API、数据库、Redis 状态。
- `alembic upgrade head` 可执行。
- `pytest` 可通过。
- OpenAPI 页面可访问。
- Web 工程可启动开发服务器并完成基础构建。
