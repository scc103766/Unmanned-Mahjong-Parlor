# 后续开发路线图

## 文档定位

本文基于当前 BMAD 产物和 `../24h_qipaishi` 参考小程序的能力盘点，作为 Python 后端 + 网页端的后续开发执行计划。已有文档已经明确产品范围、架构、领域模型和 API 分组；本文进一步回答三个问题：

- 先做什么，后做什么。
- 每个阶段交付到什么程度才算完成。
- 如何在功能对齐参考项目的同时，把 Python 版做得更可靠、更可运营。

## 当前基线判断

参考项目已经覆盖无人棋牌室的主要业务闭环：

- 顾客：登录、选店、选房、预约、支付、订单详情、开门、开电、续费、取消、换房、余额、优惠券、团购码。
- 商家：门店、房间、价格、不可预约时段、设备、订单、会员、优惠券、充值、提现、统计。
- 保洁：任务列表、接单、开始、开门、上传凭证、完成、投诉、结算、统计。
- 设备：店门、房间门、智能锁、电源、云喇叭、蓝牙锁兜底。

Python 版不能只复刻接口，需要把参考项目里散落在小程序页面中的关键规则上收到服务端：

- 多租户隔离不能只相信前端传入的 `tenant-id`。
- 房间库存、价格计算、订单状态、设备权限必须由后端裁决。
- 支付回调、退款、余额、优惠券、团购码、设备命令必须具备幂等和审计。
- 订单结束、保洁任务、房态变化需要事件化，避免人工串流程。
- Web 运营后台和平台运营需要看到异常、审计、补偿入口和基础经营数据。

目标产品形态调整为 Web-first：

- 顾客 H5：顾客选店、选房、下单、支付、开门、续费、取消。
- 保洁移动网页：任务列表、接单、开门、凭证、完成和结算记录。
- 商家 Web 后台：门店、房间、价格、设备、订单、保洁、财务和统计。
- 平台 Web 后台：租户、权限、设备供应商、异常、审计和人工补偿。
- App：业务稳定后的后续移植方向。
- 参考小程序：作为业务参考和可选兼容对象，不作为近期主产品形态。

## 交付策略

### 第一原则

- 先跑通服务端核心闭环，再扩展管理端体验。
- 网页端是近期正式产品形态，服务端 API 必须优先满足 Web/H5 体验。
- 先接 mock provider，再接真实微信支付和硬件供应商。
- 先保证库存、支付、设备、保洁四条高风险链路正确，再扩展营销和统计。
- API 使用 `/api/v1/**` 做网页端新契约，必要时通过 `/compat/member/**` 兼容参考小程序路径。
- 每个阶段都要有可运行服务、迁移脚本、自动化测试和 OpenAPI 契约。

### 推荐技术落地

| 层次 | 方案 |
| --- | --- |
| 前端 | Vue 3 + Vite + TypeScript，移动 H5 和后台分布局实现 |
| API | FastAPI + Pydantic v2 |
| ORM | SQLAlchemy 2.x |
| 数据库 | PostgreSQL |
| 迁移 | Alembic |
| 缓存/幂等/短锁 | Redis |
| 异步任务 | Celery 起步，后续可评估 Arq |
| 测试 | pytest + httpx + factory-boy |
| 代码质量 | ruff + mypy + pre-commit |
| 对象存储 | S3 兼容接口，开发期 MinIO |
| 部署 | Docker Compose 起步，后续支持多实例 API + 独立 Worker |

## Sprint 拆解

### Sprint 0：项目启动与工程底座

目标：让团队可以稳定启动服务、跑测试、建表和查看 OpenAPI。

任务：

1. 初始化项目结构：`app/`、`tests/`、`alembic/`、`scripts/`、`docs/`。
2. 建立依赖管理和基础命令：安装、测试、lint、mypy、迁移、启动。
3. 创建 FastAPI 应用入口、统一配置、结构化日志和 request_id。
4. 接入 PostgreSQL、Redis、SQLAlchemy、Alembic。
5. 实现 `GET /health` 和 `GET /api/v1/meta/version`。
6. 建立统一响应、错误码、异常处理和 OpenAPI tag 分组。

验收：

- 本地一条命令可启动 API。
- `GET /health` 返回 API、数据库、Redis 状态。
- `alembic upgrade head` 可执行。
- `pytest` 至少覆盖健康检查和配置加载。
- OpenAPI 页面可访问。

### Sprint 1：租户、用户、权限

目标：所有后续业务都运行在可信的数据作用域内。

任务：

1. 建立 Tenant、TenantApp、User、Role、UserRole、UserStoreScope。
2. 实现微信登录 mock provider 和 provider 接口抽象。
3. 实现 JWT 登录态、当前用户依赖、租户解析依赖。
4. 建立平台、商家、店员、保洁、顾客、客服角色。
5. 所有业务查询基础 repository 默认携带 `tenant_id`。
6. 设计审计日志基础表和审计写入工具。

验收：

- 不同租户数据不能互相访问。
- 无角色或无门店作用域用户无法操作门店资源。
- 登录态中的租户和角色来自服务端绑定，而不是直接信任前端 header。
- 权限失败有统一错误码和 request_id。

### Sprint 2：门店、房间、价格配置

目标：商家可以配置一个可被顾客预约的门店和房间。

任务：

1. 实现 Store：名称、地址、坐标、公告、客服、Wi-Fi、图片、营业参数。
2. 实现 Room：房间类型、容量、标签、排序、图片、状态。
3. 实现 RoomPriceRule：普通价、工作日价、通宵价、最小时长、提前预约限制。
4. 实现 RoomBlockedSlot：不可预约时段。
5. 实现设备绑定草案：房间可以关联门、锁、电源、云喇叭设备占位。
6. 对齐参考接口语义，给出房间列表、详情和价格摘要。

验收：

- 商家可创建门店、房间和价格规则。
- 顾客端可读取门店详情、房间列表和价格摘要。
- 价格和禁用时段变更写审计。
- 同一租户内门店和房间数据完整隔离。

### Sprint 3：可用性、价格试算、订单状态机

目标：完成预约下单的核心交易链路。

任务：

1. 实现 Availability Service：营业参数、禁用时段、有效订单、清洁占用综合计算。
2. 实现 Pricing Service：普通价、工作日价、通宵价、最小时长和明细解释。
3. 实现 RoomTimeLock：待支付锁、已支付占用、过期释放。
4. 实现 Order：预订单、正式订单、取消、开始、完成。
5. 实现续费试算和续费提交。
6. 实现换房试算和同事务换房。
7. 增加并发下单测试和状态机测试。

验收：

- 同一房间同一时间段不能重复占用。
- 待支付订单超时可释放时段。
- 续费只检查订单后续增量时段。
- 换房必须在同一事务内释放旧锁并创建新锁。
- 价格返回明细，可解释给顾客和客服。

### Sprint 4：支付、余额、优惠券、团购码

目标：资金与优惠形成闭环，且可幂等、可对账。

任务：

1. 实现 Payment：预下单、支付单、支付回调、查单、关闭订单。
2. 先接 mock 微信支付，再预留真实微信支付 provider。
3. 实现支付回调事件表和幂等处理。
4. 实现 Refund：申请、回调、部分退款、金额上限。
5. 实现 WalletAccount、WalletLedger、充值规则和赠送余额。
6. 实现 CouponTemplate、Coupon：领取、锁定、使用、退回。
7. 实现 GroupBuyCode：验证、锁定、核销、退回。

验收：

- 支付回调重复进入只确认一次订单。
- 支付成功但订单确认失败时有补偿任务。
- 退款总额不能超过可退金额。
- 余额、优惠券、团购码所有变化均有流水或核销记录。
- 前端支付成功回跳不能作为最终订单状态依据。

### Sprint 5：设备命令与受控开门

目标：顾客、商家、保洁的开门开电全部通过后端授权和审计。

任务：

1. 实现 Device：门店大门、房间门、智能锁、电源、云喇叭。
2. 实现 DeviceCommand：命令、操作者、业务对象、状态、请求响应、失败原因。
3. 实现设备供应商接口和 mock adapter。
4. 实现顾客订单授权开店门、房间门、智能锁、开电。
5. 实现商家授权开门、关门、测试设备。
6. 实现命令重试、短时间去重和命令状态查询。
7. 预留蓝牙锁兜底结果回写接口。

验收：

- 非订单用户、过期订单、已取消订单不能开门。
- 保洁、商家、客服开门走不同权限路径，均写审计。
- 每次设备操作都有命令记录和可追踪结果。
- 设备 adapter 可被自动化测试替换。

### Sprint 6：保洁任务闭环

目标：订单结束后自动生成清洁任务，保洁履约和房态联动。

任务：

1. 实现订单完成、取消且已使用、人工触发后的保洁任务生成。
2. 实现 CleaningTask：待接单、已接单、进行中、待审核、已完成、已驳回、已取消、已结算。
3. 实现保洁任务列表、接单、取消接单、开始、完成。
4. 实现保洁任务授权开门。
5. 实现 CleaningProof 图片凭证和备注。
6. 实现商家审核、投诉、驳回、结算。
7. 实现房间清洁状态对可预约性的影响。

验收：

- 一个订单不会重复生成多个有效保洁任务。
- 保洁员只能操作自己授权或可接范围内的任务。
- 保洁开门必须校验任务、房间和授权时间窗口。
- 房间是否允许未清洁继续预约可按门店配置控制。

### Sprint 7：商家运营、提现、统计

目标：商家可完成日常经营，平台可发现异常。

任务：

1. 实现后台订单处理：取消、退款、改时、续费、换房、人工完成。
2. 实现会员列表、消费记录、优惠券发放。
3. 实现充值规则管理和充值统计。
4. 实现 Withdrawal：申请、审核、打款确认、驳回。
5. 实现收入、订单、退款、充值、房间使用率、保洁统计。
6. 实现平台异常列表：支付异常、设备失败、退款异常、任务积压。
7. 增加审计查询和人工补偿入口的 API。

验收：

- 商家常见经营操作都有 API、权限和审计。
- 统计口径可追溯到订单、支付、退款、充值、提现、清洁事实表。
- 平台可以按租户、门店、时间筛选异常。

### Sprint 8：网页端 MVP

目标：让正式产品形态从 API 进入可访问、可操作的网页端。

任务：

1. 创建 Vue/Vite/TypeScript Web 工程、环境配置、路由、主题、API client、登录态。
2. 顾客 H5 完成选店、选房、价格试算、下单、支付 mock、订单详情、开门 mock。
3. 保洁移动网页完成任务列表、接单、开始、开门、上传凭证、完成。
4. 商家 Web 后台完成工作台、门店房间、价格、设备、订单、保洁、统计基础页。
5. 平台 Web 后台完成租户、角色、异常、审计和设备供应商基础页。
6. 针对移动网页和后台桌面页面分别做布局、表格密度、导航和权限入口。

验收：

- 顾客 H5 可完成选房、下单、支付 mock、开门 mock 和订单管理。
- 保洁移动网页可完成接单、开门 mock、凭证上传和任务完成。
- 商家 Web 后台可完成日常经营配置和订单处理。
- 平台 Web 后台可查看异常和审计。
- Web 构建产物可部署。

### Sprint 9：网页发布、兼容层、试运营

目标：让网页端具备真实联调和试运营条件。

任务：

1. 实现 `/compat/member/**` 兼容层的高频接口映射，作为参考小程序迁移辅助。
2. 梳理参考小程序页面所需字段，建立字段兼容表。
3. 对接真实微信登录、微信支付和至少一种硬件供应商。
4. 完成顾客 H5、保洁移动网页、商家后台、平台后台、后端、支付、设备的端到端联调。
5. 做并发下单、支付回调重复、设备失败、保洁任务积压测试。
6. 完成 Web 构建、静态资源部署、反向代理和环境配置。
7. 配置结构化日志、关键指标、告警和异常补偿后台入口。
8. 输出上线清单、回滚方案、试运营问题登记模板。

验收：

- 测试门店可通过顾客 H5 完成选房、下单、支付、开门、结束、保洁闭环。
- 商家可通过 Web 后台完成配置、订单处理、设备管理、统计和异常处理。
- 参考项目核心接口语义均有新接口或兼容映射。
- 高风险链路具备审计、幂等、告警和人工补偿路径。
- 试运营数据可清理、可回放、可对账。

## 模块依赖顺序

```text
M0 工程底座
  -> M1 租户/用户/RBAC/审计
    -> M2 门店/房间/价格/禁用时段
      -> M3 可用性/库存锁/订单状态机
        -> M4 支付/余额/优惠/退款
        -> M5 设备/命令/开门开电
          -> M6 保洁任务/房态联动
            -> M7 商家运营/统计/提现
              -> M8 网页端 MVP
                -> M9 Web 发布/兼容联调/试运营
```

## 上线前检查清单

## 主要风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 硬件协议不完整 | 真实开门联调延期 | 先做设备 adapter 和 mock command，尽早收集供应商文档 |
| 微信支付配置未就绪 | 支付闭环无法实测 | Sprint 4 先用 mock provider，真实 provider 独立切换 |
| 房间库存超卖 | 直接造成经营事故 | 使用数据库事务和排他约束，不依赖应用层查询判断 |
| 兼容旧小程序字段复杂 | 联调成本上升 | 新 API 服务网页端，兼容层只做字段转换和路径适配 |
| 移动网页与后台体验差异大 | 一个界面难以同时好用 | 顾客/保洁移动页面和商家/平台后台分别设计布局 |
| 后续 App 迁移成本 | 二次开发工作量上升 | 先稳定 API、流程和字段口径，再迁移 App |
| 统计口径反复变化 | 影响后台交付 | 先保留事实表，聚合表后置，可按口径重算 |
| 多租户隔离遗漏 | 数据安全风险 | Repository、权限依赖、测试用例三层同时约束 |
