# 开发计划

## 产品范围

Python 版系统需要实现参考项目的完整业务闭环，并补齐后端系统能力：

- 顾客 H5/移动网页：顾客预约、支付、订单和开门。
- 保洁移动网页：保洁任务、开门、凭证和结算记录。
- 商家 Web 后台：商家经营配置、订单、设备、保洁、财务和统计。
- 平台 Web 后台：租户、权限、设备供应商、异常和审计。
- 顾客预约支付开门闭环。
- 商家门店、房间、价格、设备、订单、营销和统计管理。
- 保洁任务生成、接单、开门、完成、审核和结算。
- 平台多租户、权限、设备供应商、支付配置、异常和审计。
- 可靠的支付幂等、库存锁、设备命令追踪和数据隔离。

| --- | --- | --- | --- |

## 阶段验收

### M0 项目基础

- 服务可通过 `uvicorn` 或容器启动。
- 数据库迁移可创建基础表。
- 测试命令可运行，并至少包含健康检查测试。
- OpenAPI 文档可访问。
- 配置支持 dev/test/prod 分环境。

### M1 租户与权限

- 支持平台、租户、商家、店员、保洁员、顾客的身份模型。
- 支持 JWT 登录态、角色权限和门店作用域。
- 所有业务查询默认受 `tenant_id` 和门店授权约束。
- 高风险操作具备审计日志基础能力。

### M2 门店房间配置

- 支持租户下多门店和多房间。
- 门店可配置公告、地址、坐标、客服、Wi-Fi、图片、清洁参数。
- 房间可配置分类、标签、图片、普通价、工作日价、通宵价、最小时长、提前预约和禁用时段。

### M3 预约与订单

- 任意房间同一时间段不能被重复锁定。
- 价格计算支持普通价、工作日价、通宵价、最小时长、优惠抵扣前置校验。
- 订单状态机覆盖待支付、已支付、进行中、已完成、已取消、退款中、已退款。
- 续费和换房必须重新校验库存和价格。

### M4 支付与资金

- 微信支付下单、回调、查单、退款接口具备幂等能力。
- 支持余额支付和赠送余额流水。
- 优惠券、团购码只能在合法订单和合法状态下使用。
- 所有资金变化有流水和审计。

### M5 设备控制

- 开门、开锁、开电全部通过后端权限校验。
- 每次设备操作生成命令记录，保存请求、响应、状态、失败原因和操作者。
- 设备供应商通过适配器隔离，至少有一个模拟适配器用于测试。
- 支持订单授权、商家授权、保洁任务授权三种开门场景。

### M6 保洁闭环

- 订单结束、取消或人工触发后可生成清洁任务。
- 保洁员可接单、开始、开门、上传凭证、完成。
- 商家可审核、驳回、投诉、结算。
- 房间是否允许未清洁继续预约可按门店配置控制。

### M7 商家运营

- 商家可处理订单取消、退款、改时、续费、换房。
- 支持会员、优惠券、充值规则和提现记录。
- 提供订单量、收入、退款、充值、房间使用率、保洁数据等基础统计。
- 平台运营可查看跨租户异常和审计。

### M8 网页端

- 顾客 H5 可完成选店、选房、下单、支付、开门、续费和取消。
- 保洁移动网页可查看任务、接单、开门、上传凭证和完成任务。
- 商家 Web 后台可管理门店、房间、价格、设备、订单、会员、营销、财务和统计。
- 平台 Web 后台可管理租户、权限、设备供应商、异常和审计。
- 网页端构建产物可部署到静态服务或同域网关。

## 功能增强清单

| 增强点 | 参考项目现状 | Python 版策略 |
| --- | --- | --- |
| 租户隔离 | 前端传 `tenant-id` | 服务端根据登录态、应用绑定和数据作用域校验 |
| 价格计算 | 页面参与较多 | 服务端统一 Pricing Service |
| 库存锁 | 后端已有能力但前端不可见 | 独立 Availability 模块和数据库排他约束 |
| 支付幂等 | 需要明确化 | 支付单、回调事件和幂等键 |
| 设备控制 | 多接口直连 | 命令表 + 适配器 + 审计 + 重试 |
| 保洁任务 | 业务覆盖较完整 | 订单事件驱动生成，任务与房态联动 |
| API 契约 | 页面散落 | OpenAPI + 版本化路由，供 Web 和后续 App 使用 |
| 客户端形态 | 微信小程序为主 | 先做顾客 H5、保洁移动网页、商家/平台 Web 后台，后续移植 App |
| 测试 | 小程序侧难覆盖 | 单元、API、状态机、并发下单、Web 集成测试 |

## 关键风险

- 硬件供应商协议不完整会阻塞真实开门联调，需要先做模拟适配器和接口抽象。
- 微信支付和退款需要真实商户配置，开发期应提供沙箱或 mock provider。
- 房间库存锁如果只靠应用层判断会出现并发超卖，必须落到数据库约束或事务锁。
- 小程序旧接口如果完全保留路径，后端会背负历史命名；建议把兼容层作为迁移辅助，网页端直接使用新领域 API。
- 网页端移动页面和后台页面的信息密度差异大，必须分别设计布局和关键流程。
- 统计口径容易反复变化，应先以事实表为基础，后续再做聚合表和缓存。

## 推荐首轮开发顺序

1. 创建项目脚手架、配置、健康检查、数据库迁移和测试。
2. 建立租户、用户、角色、门店、房间核心表。
3. 实现房间可用性查询和价格计算。
4. 实现预订单、锁定时段、订单状态机。
5. 加入支付 mock 和回调幂等。
6. 加入设备 mock 适配器，完成订单授权开门。
7. 由订单结束事件生成保洁任务。
8. 创建 Web 前端工程，先跑通登录、门店列表、订单详情和 mock 开门。
